# Правила проекта Clash Clan Analytics

Этот файл задаёт постоянные правила для любого нового сеанса Codex.

## Границы workspace

- Workspace root: `D:\coc`.
- Единственный Git root: `D:\coc\repo`.
- Код, техническая документация и публичные файлы сайта находятся в `D:\coc\repo`.
- Результаты конкретных запусков сохраняются только в `D:\coc\runs` и не добавляются в Git.
- Постоянные локальные данные находятся в `D:\coc\data` и не добавляются в Git.
- Секреты находятся только в `D:\coc\data\secrets`.
- Временные файлы, логи и локальная автоматизация находятся в `D:\coc\local`.
- Без явного разрешения запрещены любые изменения вне `D:\coc`.
- `D:\work` и `D:\study` разрешено использовать только как read-only примеры организации. Нельзя переносить из них рабочие секреты, подключения, код, базы или пользовательские данные.

## Безопасность и разрешения

- Секреты, API-токены, `.env`, локальные базы, локальные конфиги и внутренние заметки никогда не добавляются в Git.
- Доступ к Clash of Clans API требует отдельного явного разрешения.
- Создание API-ключа требует отдельного явного разрешения.
- Любой сетевой запрос требует отдельного явного разрешения.
- Каждый Git commit требует отдельного явного разрешения.
- Git push, создание GitHub-репозитория и публикация GitHub Pages требуют отдельных явных разрешений.
- Планировщик Windows и любые автоматические запуски требуют отдельного явного разрешения.
- Для опасных, массовых или необратимых действий сначала составить точный план, определить цели и резервную копию, затем запросить отдельное разрешение.
- Полную историю нельзя удалять, очищать или перезаписывать без резервной копии и отдельного разрешения.

## Данные и публикация

- Исходные данные, расчётные данные и опубликованные данные должны быть логически разделены.
- Публичные данные сайта и внутренние данные руководства должны быть разделены на уровне файлов, экспортов и полей.
- Нельзя публиковать внутренние комментарии, уважительные причины пропусков, предупреждения, дисциплинарные пометки и служебные статусы.
- В публичных материалах не должна формироваться «доска позора»; показатели должны иметь нейтральный контекст.
- В Git допустимы только специально подготовленные публичные выгрузки, например JSON/CSV в `site/data`.

## Метрики и идентичность игроков

- Каждая метрика должна иметь документированное определение, источник данных, ограничения и версию логики.
- Изменение логики метрики сопровождается новой версией расчёта и записью решения; исторические результаты не перезаписываются молча.
- Игроки связываются по стабильному `player_tag`, а не по отображаемому имени.
- Проектные гипотезы не становятся правилами клана без согласования руководства.

## Порядок работы

1. Сначала прочитать этот файл и релевантную документацию.
2. Проверить границы задачи и необходимые отдельные разрешения.
3. Хранить локальные входы в `data`, результаты запусков в `runs`, а публикуемый код и документацию — в `repo`.
4. Перед публикацией проверять экспорт на секреты, персональные и внутренние поля.
5. Перед Git-операциями проверять `git -C D:\coc\repo status` и отсутствие случайных локальных данных.

## Codebase Memory

- Project: `D-coc`; workspace root: `D:/coc`.
- Ignore file: `D:\coc\.cbmignore`.
- Refresh script: `D:\coc\repo\scripts\workspace\refresh_codebase_memory.ps1`.
- Запуск требует отдельного явного разрешения; одно разрешение означает один запуск, автоматические повторы запрещены.
- Скрипт работает только с существующим проектом. Создавать новый проект через refresh-скрипт запрещено.
- Перед запуском обязательны точная проверка имени и root, наличие полного `.cbmignore` и полностью чистый Git status, включая отсутствие untracked и staged-файлов.
- Codebase Memory не должна изменять файлы workspace, Git index или HEAD.
- `data`, `runs`, `local`, `.git`, `.codebase-memory` и `repo/site/data` индексировать запрещено.
