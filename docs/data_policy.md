# Политика данных

## 1. Публичные данные

Согласованные сведения о составе, нейтральные показатели активности и игровые агрегаты, предназначенные для статического сайта. Публикуются только после фильтрации и проверки.

Для Игр кланов в публичный слой могут входить отображаемое имя, факт участия, очки, количество участий, средние очки, количество максимумов и агрегаты за последние 12 месяцев и всё время. `player_tag` публикуется только после отдельного разрешения руководства.

### Public roster allowlist v1

Draft-allowlist состава разрешает только:

- `display_name`;
- отображаемое название должности из внутреннего `role`;
- `town_hall_level` при наличии валидного значения;
- безопасное агрегированное `activity_summary` после утверждения формулы;
- `last_regular_war_date`, если дата известна;
- `verification_status` с нейтральным пользовательским названием.

`player_tag` внутренне необходим для устойчивой связи снимков, но его публикация не согласована, поэтому в allowlist он не входит. Отображаемое имя не является устойчивым ключом.

Публикуется только поле, явно включённое в allowlist. Отсутствие поля в denylist не означает разрешение публикации. Полный draft-контракт: [roster_data_contract.md](roster_data_contract.md).

## 2. Внутренние данные руководства

Комментарии, уважительные причины пропусков, предупреждения, решения о составе, детали формирования статуса проверки и другая управленческая информация. Хранятся только локально и не входят в публичные экспорты или Git. В публичный allowlist может войти только нейтральное агрегированное состояние `verification_status`, без причин и истории решений.

Для Игр кланов к внутренним данным относятся причина отсутствия, комментарии руководства, отметка об уважительности причины, служебная классификация спорных случаев и история ручных исправлений. Отсутствие нельзя публично сопровождать причиной без отдельного решения и согласия.

Для состава запрещены к публичному экспорту `player_tag` до отдельного решения, внутренние идентификаторы, причины отсутствия, спорные ситуации, заметки руководства, личные комментарии, причины выбора состава, внутренняя оценка надёжности, история ручных корректировок, timestamps импорта и raw API payload. Свободное текстовое поле с оценкой игрока не входит в публичную карточку.

### Личные и управленческие заметки

Заметки главы, соруководителей и старейшин, спорные ситуации, личные замечания, причины решений, обстоятельства выбора состава, внутренняя оценка надёжности и история ручных корректировок являются чувствительными проектными данными.

- Доступ к ним допускается только локально.
- Их нельзя публиковать, индексировать Codebase Memory, добавлять в Git или копировать в Obsidian.
- Причина или оценочный комментарий не отображаются на публичной странице.
- Публичные агрегаты не должны позволять восстановить личный комментарий или причину решения.
- Предварительное место хранения – `D:\coc\data\manual\leadership_notes` либо будущая локальная SQLite; реальные записи на текущем этапе не создаются.

## 3. Технические данные

Raw JSON, SQLite, промежуточные расчёты, run-манифесты, логи, диагностика и резервные копии. По умолчанию локальны. В Git допускаются только обезличенные тестовые фикстуры, если их создание и проверка будут отдельно согласованы.

Технические поля Игр кланов включают идентификатор события, дату импорта, источник, версию файла, статус валидации и версию логики расчёта. Они обеспечивают воспроизводимость и не становятся публичными автоматически.

### Показатели звёзд

- `clan_stars` является официальным публичным агрегатом из `current war -> clan.stars`.
- `attack_stars_total` является рассчитанной технической суммой результатов атак и не подменяет официальный счёт.
- `stars_earned` относится к результатам атак конкретного игрока, а не обязательно к чистому приросту счёта клана.
- `new_stars_contributed` является inferred metric. Она допустима только при полном уникальном глобальном `attack.order` и известных defender links; иначе публикуется `null`.

Player tags и defender tags используются для внутреннего связывания и расчётов, но не входят в публичный JSON.

Public privacy scanning normalizes camelCase, snake_case and separator variants of private keys. It also rejects Windows drive paths, UNC paths, raw source references, authorization/token values and DPAPI metadata. Safe HTTPS URLs and neutral public text remain allowed.

## 4. Секреты

API-токены, ключи и данные доступа. Хранятся только в `D:\coc\data\secrets`, не копируются в документацию, логи, Git или сайт.

## Принципы публикации

- Публичная часть не должна становиться «доской позора».
- Метрики показываются нейтрально, с определением и контекстом ограничений.
- Внутренние причины и дисциплинарные оценки не выводятся наружу.
- Экспорт строится по разрешённому списку полей, а не удалением известных секретных полей.
- Публиковать можно только явно разрешённые поля; отсутствие поля в denylist не делает его публичным.
- `player_tag` используется для связи записей; отображаемое имя не является стабильным ключом.
- История сохраняется; исправления и изменения логики версионируются.

### Manual evidence для старых войн

Локальный overlay `linked_manual_evidence.json` остаётся отдельным от authoritative API history. Штатная сборка может безопасно добавить в public history только разрешённую проекцию screenshot evidence: coverage, нейтральные counts, позиции на карте и unresolved source conflicts. Manual facts не меняют API observations, canonical attacks, lifecycle или reconciliation. Локальные пути, hashes, evidence IDs, tags, order и provenance metadata не публикуются. Отсутствующий или невалидный overlay не останавливает updater: сайт продолжает API-only publication.
