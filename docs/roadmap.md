# Roadmap

Каждый этап начинается только после проверки результатов предыдущего. Указанные разрешения не считаются выданными этим документом.

## Product roadmap

Пользовательская последовательность описывает, что посетитель сможет увидеть в продукте. Локальный UI-прототип продуктового этапа может появиться раньше, чем завершена техническая интеграция его данных.

1. **Информационная страница – завершено.**
2. **Публикация GitHub Pages – завершено.**
3. **Сбор обратной связи клана – текущий этап.**
4. Корректировка оформления и структуры – после обратной связи.
5. Состав клана.
6. Карточки игроков.
7. Обычные КВ.
8. Остальные аналитические модули.

Реальные данные и их автоматическое получение ещё не подключены. Дальнейшая продуктовая разработка приостановлена до завершения первого раунда обратной связи клана.

Локальный интерфейс заметок руководства является отдельной будущей задачей и не входит в публичный GitHub Pages prototype.

## Technical roadmap

Техническая последовательность обеспечивает данные для продуктовых этапов:

1. Безопасный API probe либо подтверждение ручного источника.
2. Воспроизводимый snapshot.
3. Локальная SQLite.
4. Валидируемые импорты.
5. Версионированные расчёты.
6. Публичный экспорт через allowlist.
7. GitHub Pages.
8. Автоматизация.

Подготовленный draft-контракт состава определяет модель и границы будущего экспорта, но не подключает источник и не заменяет технические этапы snapshot, импорта и валидации.

Техническая реализация состава приостановлена до завершения первого раунда обратной связи. После него источник состава выбирается отдельно и не считается заранее определённым.

## Подробный technical roadmap

Таблица ниже детализирует инфраструктурные этапы, разрешения и критерии завершения. Номера технических этапов не являются номерами пользовательских экранов или модулей.

| Этап | Цель | Ожидаемый результат | Отдельное разрешение | Критерий завершения |
|---|---|---|---|---|
| 0 – структура workspace | Разделить код, данные, runs, заметки и local | Каталоги, правила, документация, локальный Git | `git init` разрешён текущим bootstrap-заданием | Структура и проверки завершены, commit отсутствует |
| 1 – проверка API | Проверить минимальный официальный запрос | Локальный API probe и документированный ответ | Сеть, создание ключа, API-доступ | Ответ получен безопасно, секрет не раскрыт |
| 2 – первый снимок состава | Сохранить воспроизводимый snapshot | Raw JSON и локальный нормализованный результат | API и сеть, если не разрешены отдельно | Состав связан по `player_tag`, run проверен |
| 3 – локальная SQLite | Создать локальную модель истории | Схема, миграция, импорт snapshot | Создание/изменение локальной БД | Повторный импорт предсказуем, база вне Git |
| 4 – пожертвования | Считать изменения за период | Версионированная метрика и тесты | API-запросы/публикация при необходимости | Определение согласовано, тесты проходят |
| 5 – обычные КВ | Импортировать войны и участия | Локальная история войн и атак | API, включая журнал войн | Доступность подтверждена, импорт воспроизводим |
| 6 – метрики атак | Рассчитать зеркала, звёзды, добивания, пропуски | Каталог формул, версии и тесты | Согласование руководства | Все определения утверждены и протестированы |
| 7 – очередь 50/50 | Подготовить рекомендательную очередь | Объяснимый локальный список-кандидат | Утверждение правил руководством | Рекомендации воспроизводимы и не выдаются за решение |
| 8 – ЛВК | Добавить отдельную модель ЛВК | История и показатели ЛВК | API и согласование отличий правил | ЛВК не использует 50/50 автоматически |
| 9 – рейды | Добавить рейдовые выходные | История участия и согласованные показатели | API и согласование метрик | Импорт и расчёты проверены |
| 10a – Игры кланов: исследование источника | Проверить официальный API и другие допустимые источники без предположений | Зафиксированный перечень доступных полей и ограничений | Сеть, API-доступ и probe | Источник подтверждён доказательствами либо явно выбран ручной ввод |
| 10b – Игры кланов: ручной импорт v1 | Определить CSV-контракт, валидировать события и `player_tag`, предотвращать дубли | Воспроизводимый локальный импорт с историей исправлений | Реализация кода, создание локального хранилища и запуск импорта | Повторный импорт не создаёт дубли, ошибки и исправления отслеживаются |
| 10c – Игры кланов: метрики | Рассчитать участия, первое/последнее участие, очки, среднее, максимумы и период 12 месяцев | Версионированные метрики; отсутствие и причина отделены во внутренний слой | Согласование формул, статусов и периода руководством | Формулы протестированы; пропуски не входят нулём в среднее, регулярность считается отдельно |
| 10d – Игры кланов: публичное отображение | Показать последнее событие, агрегаты за 12 месяцев и всё время, данные в карточке игрока | Безопасное публичное представление без причин отсутствия | Утверждение публичных полей и публикации `player_tag` | Публичный экспорт проверен и не содержит внутренних причин или комментариев |
| 11 – статический сайт | Создать публичное представление | Локально собранный сайт с публичными данными | Утверждение публичных полей | Сайт работает без сети и не содержит внутренних данных |
| 12 – GitHub Pages | Опубликовать статический сайт | Доступный публичный URL | GitHub repo, remote, commit, push, Pages | Выполнено: Pages использует artifact `site`, автоматический deployment после push настроен |
| 13 – автоматизация | Автоматизировать локальное обновление | Надёжное задание с логами и обработкой ошибок | Планировщик, автозапуск и любые Git-действия | Тестовый цикл успешен и имеет безопасный откат |
| L1 – локальные заметки руководства | Спроектировать непубличное хранение и локальный интерфейс вне сайта | Отдельная локальная модель с контролем доступа и историей исправлений | Согласование полей, создание локальной БД и реализация интерфейса | Данные остаются вне Git, Codebase Memory, Obsidian и публичного экспорта |
