# Draft-контракт состава клана

Status: `draft`
Version: `roster_v1_draft`
Real data source: `TBD`

## Назначение

Документ определяет границу между будущим источником состава, внутренней нормализованной моделью и публичным представлением. Он фиксирует устойчивую внутреннюю идентификацию игрока, поля источника и расчётов, публичный allowlist, запрещённые для экспорта данные, представление отсутствующих данных и будущую связь с сайтом.

Контракт не подключает источник и не содержит реальных участников. До отдельного решения никакие данные состава не импортируются и не публикуются.

`player_tag` является внутренним устойчивым ключом. `display_name` может меняться и не используется для дедупликации, связи снимков или истории.

Свободного публичного текстового поля с оценкой игрока в модели нет. Внутренние комментарии, причины и решения не должны смешиваться с перечисленными полями.

## Значения `role`

Внутренние значения:

```text
member
elder
co_leader
leader
```

Русские названия формируются только на уровне представления. Отображаемое название роли не сохраняется как внутреннее значение и не используется как ключ.

## Значения `verification_status`

```text
not_evaluated
candidate
verified
extended_evaluation
review_required
```

Это статус участия в обычных КВ, а не оценка личности или общий рейтинг. Он не должен автоматически влиять на публичную оценку игрока или другие направления активности. Статус `verified` возможен после двух обычных КВ без спорных ситуаций; при вопросах проверка продлевается ещё на одну или две войны. Спорные ситуации не классифицируются автоматически.

На публичном уровне значения получают нейтральные пользовательские названия без публикации причин, обсуждений или внутренних комментариев.

## Значения `activity_summary`

Допустимые будущие значения:

```text
active
limited_data
inactive
unknown
```

Окончательная формула не определена и требует отдельного согласования. `unknown` означает недостаточность данных. Отсутствие данных нельзя подменять значением `inactive`; `inactive` допустимо только после утверждения формулы и при наличии достаточных подтверждённых данных.

## Представление отсутствующих данных

- Неизвестная необязательная дата представляется как `null`.
- Недостаточность данных для агрегата активности представляется как `unknown`, а не `inactive`.
- Отсутствие первого снимка не означает, что игрок вышел из клана. `is_current_member: false` допустим только при подтверждённом источником состоянии.
- Пустой подтверждённый состав в публичном формате представляется как `members: []`.
- До подключения подтверждённого источника сайт показывает общее пустое состояние и не создаёт фиктивных записей.

## Public roster allowlist v1

На уровне draft в будущий публичный состав могут входить только:

- `display_name`;
- отображаемое русское название должности, построенное из `role`;
- `town_hall_level` при наличии валидного значения;
- безопасное агрегированное `activity_summary` после утверждения формулы;
- `last_regular_war_date`, если дата известна;
- `verification_status` с нейтральным пользовательским названием.

**Публикуется только поле, явно включённое в allowlist.** Отсутствие поля в denylist не означает разрешение публикации.

Публикация `player_tag` не согласована. Поле не входит в `Public roster allowlist v1` и по умолчанию не записывается в публичный файл.

## Поля, запрещённые для публичного экспорта

- `player_tag` до отдельного решения;
- `exp_level`, `clan_rank`, `previous_clan_rank`;
- `donations`, `donations_received`, `trophies`, `builder_base_trophies`;
- внутренние идентификаторы;
- причины отсутствия;
- данные и описание спорных ситуаций;
- заметки руководства;
- личные и оценочные комментарии;
- причины назначения или неназначения на войну;
- внутренняя оценка надёжности;
- история ручных корректировок;
- технические timestamps импорта, включая `roster_snapshot_at`;
- сырые API payload;
- секреты, токены и технические данные доступа.

Эти данные не должны попадать в Git, Obsidian, публичный JSON или HTML. Наличие внутреннего поля не создаёт разрешения на его публикацию.

## Будущий публичный формат

Планируемый генерируемый файл:

```text
site/data/roster.json
```

Файл на текущем этапе не создаётся. Draft-схема:

```json
{
  "generated_at": "ISO-8601 timestamp",
  "source_status": "confirmed",
  "members": [
    {
      "display_name": "string",
      "role": "member",
      "town_hall_level": 0,
      "activity_summary": "unknown",
      "last_regular_war_date": null,
      "verification_status": "not_evaluated"
    }
  ]
}
```

Это только схема, а значения не описывают реального участника. `town_hall_level: 0` является placeholder схемы и не допускается в реальном экспорте. Пустой состав представляется как `members: []`.

Внутренняя и публичная модели различаются. Публичный JSON формируется только через allowlist после валидации; `player_tag` не попадает в него до отдельного решения. `generated_at` является метаданными публичного файла, а внутренние timestamps отдельных импортов не экспортируются.

## Источники состава

### Основной предполагаемый источник

Официальный Clash of Clans API после отдельного разрешения, создания безопасной конфигурации и минимального probe. Авторизованный официальный Swagger review от `2026-07-19` подтвердил `GET /clans/{clanTag}`, `Clan.memberList`, точные используемые wire-field names/types и Bearer JWT header. Подробности и provenance: [clash_api_data_foundation.md](clash_api_data_foundation.md) и [clash_api_authenticated_contract_2026-07-19.md](clash_api_authenticated_contract_2026-07-19.md).

Base URL, version prefix, requiredness, nullability, enum values, status codes и error mapping всё ещё не подтверждены. Наличие internal snapshot field не добавляет его в public allowlist. Настоящий response пока не получался.

### Временный резервный источник

Ручной локальный CSV или JSON из подтверждённого состава. Файлы импорта сейчас не создаются; формат и происхождение снимка должны быть согласованы отдельно.

Будущий импорт обязан проверять:

- формат `player_tag`;
- отсутствие дублей и уникальность записей по `player_tag`;
- допустимое внутреннее значение `role`;
- допустимый уровень ратуши без placeholder-значений;
- наличие и корректность snapshot timestamp;
- различие между подтверждённым выходом игрока и временно отсутствующими данными;
- отсутствие внутренних заметок и запрещённых полей в публичном наборе.

## Будущая связь с сайтом

Подтверждённый источник заполняет внутреннюю модель по `player_tag`. После валидации и расчёта разрешённых агрегатов отдельный экспорт применяет allowlist и генерирует `site/data/roster.json`. Статический сайт читает только этот подготовленный публичный файл и не получает доступ к локальной базе, raw payload или внутренним полям.
